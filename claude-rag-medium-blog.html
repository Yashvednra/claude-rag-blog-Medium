Why RAG Exists: The Context Window Problem

Every Claude conversation happens within a context window — a finite amount of text the model can see at once. Claude's context is impressively large, but no context window can hold an entire company's documentation, a law firm's case archive, or a developer's full codebase.

💡 Quick Rule of Thumb
If your knowledge base is under ~200,000 tokens (roughly 500 pages), you can simply paste the entire thing into the prompt. Claude's prompt caching feature makes this fast and affordable — reducing costs by up to 90%. But once your knowledge base grows beyond that, you need RAG.

RAG solves this by retrieving only the most relevant fragments from a large knowledge base and injecting them into Claude's prompt at query time. The model never sees the full corpus — just the parts that matter most for the current question.

Stage 1 — Text Chunking: Slicing the Knowledge Base

Before anything can be retrieved, your documents must be broken into manageable pieces. This step is called chunking, and the strategy you choose has an outsized effect on downstream quality.

Common Chunking Strategies

1
Fixed-size character splitting

Split every N characters (e.g. 512). Simple and fast, but often cuts mid-sentence or mid-thought, destroying semantic coherence.

2
Sentence or paragraph splitting

Respect natural language boundaries. Better coherence, but chunk sizes vary greatly, which can affect embedding quality.

3
Semantic chunking

Group sentences by topic shift, detected via embedding similarity. Produces the most context-rich chunks but is computationally heavier.

4
Overlapping windows

Add a sliding overlap (e.g., 50–100 token overlap between chunks) so no critical context sits exactly on a boundary split.

The Anthropic course emphasizes that there is no universally perfect chunk size — it depends on your content type, the embedding model you use, and how specific or broad user queries tend to be.

Stage 2 — Text Embeddings: Meaning as Math

Once you have chunks, each one is passed through an embedding model — a neural network that converts text into a dense vector of numbers. These vectors encode semantic meaning: chunks about similar topics land near each other in vector space, regardless of exact wording.

When a user asks a question, that question is also embedded into the same vector space. Retrieval then becomes a geometric problem: find the stored vectors closest to the query vector. This is semantic search, and it's what makes RAG feel intelligent rather than mechanical.

📐 Under the Hood
Similarity is typically measured with cosine similarity — the cosine of the angle between two vectors. A score of 1.0 means identical meaning, 0.0 means totally unrelated. The top-K chunks with the highest scores are selected for the prompt.

Stage 3 — BM25: The Old Algorithm That Still Wins at Exact Matching

Embedding-based semantic search is powerful, but it has a well-known weakness: exact keyword matching. If a user asks for "Error code TS-999," an embedding model might return results about error codes in general — while completely missing the specific chunk that contains the literal string "TS-999."

Enter BM25 (Best Matching 25), a classic lexical ranking algorithm that operates on term frequency and inverse document frequency. It finds chunks that share the exact words of the query, making it ideal for:

Error codes · Product SKUs · Names and proper nouns · Legal citations · Technical identifiers

The course teaches a Multi-Index RAG pipeline that runs both searches in parallel — BM25 for lexical precision and embeddings for semantic breadth — then merges and deduplicates results using rank fusion before injecting them into the prompt. This hybrid approach consistently outperforms either method alone.

The Context Conundrum — and Why Classic RAG Falls Short

Here is a problem that trips up many RAG implementations. Consider this chunk extracted from an SEC filing:

Retrieved Chunk — Traditional RAG
"The company's revenue grew by 3% over the previous quarter."

Which company? Which quarter? The chunk, stripped of its surrounding document context, is nearly useless. This is the context conundrum: chunking sacrifices the very context that makes a piece of information meaningful.

Traditional RAG

  • Chunks lose document context
  • Ambiguous when read in isolation
  • Higher retrieval failure rate
  • More hallucination risk

Contextual Retrieval

  • Each chunk carries its own context
  • Unambiguous and self-contained
  • 49% fewer retrieval failures
  • Significantly reduced hallucinations

Anthropic's Answer: Contextual Retrieval

Anthropic's Contextual Retrieval technique solves the context conundrum by having Claude itself annotate every chunk before it is indexed. The process uses a prompt like this:

Anthropic's Contextualization Prompt
<document>
{{WHOLE_DOCUMENT}}
</document>

Here is the chunk we want to situate within the whole document:
<chunk>
{{CHUNK_CONTENT}}
</chunk>

Please give a short succinct context to situate this chunk within
the overall document for improving search retrieval. Answer only
with the succinct context and nothing else.

Claude responds with a concise 50–100 token description of where this chunk sits within its document. That description is then prepended to the chunk before embedding. The resulting contextualized chunk looks like:

Contextualized Chunk
This chunk is from an SEC filing on ACME Corp's performance in Q2
2023; the previous quarter's revenue was $314 million.
"The company's revenue grew by 3% over the previous quarter."

Now the chunk is self-contained and unambiguous. When embedded, its vector reflects both the original content and its document context. The same contextual text is also fed into the BM25 index, boosting exact-match retrieval too.

The Full RAG Pipeline — Step by Step

1
Ingest & Chunk

Split source documents into chunks of appropriate size using a strategy suited to your content type.

2
Contextualize with Claude

Pass each chunk + its parent document to Claude. Claude prepends a 50–100 token situational context. Use prompt caching to pass the document once, not once per chunk.

3
Build Dual Indexes

Create semantic embeddings of contextualized chunks and store in a vector database. Simultaneously build a BM25 lexical index over the same chunks.

4
Retrieve at Query Time

Embed the user's query. Run both semantic search and BM25 search in parallel. Merge results with rank fusion, deduplicate, select top-K.

5
Rerank (Optional but Powerful)

Pass top-K candidates through a cross-encoder reranker for a final precision boost. This is where the biggest accuracy gains happen.

6
Augment & Generate

Inject the final retrieved chunks into Claude's prompt alongside the user query. Claude generates a grounded, contextually accurate response.

Cost Control: Prompt Caching to the Rescue

The obvious objection to Contextual Retrieval is cost: if you need to pass the entire parent document to Claude for every chunk, and you have a million chunks, doesn't that get expensive?

This is where prompt caching makes Contextual Retrieval practical. Claude can cache a document in memory across multiple API calls. You load the document into the cache once, then reference that cache for every chunk within it — avoiding redundant token processing.

💰 Cost Estimate from Anthropic
Assuming 800-token chunks, 8,000-token documents, 50-token instructions, and 100 tokens of context per chunk, the one-time cost to contextualize an entire knowledge base runs approximately $1.02 per million document tokens — a one-time preprocessing investment, not a recurring cost.

Implementing the Full RAG Flow in Python

The Anthropic course walks through a complete Python implementation. Here's a simplified sketch of the indexing phase to illustrate the architecture:

Python — Contextual RAG Indexing (Simplified)
import anthropic
from sentence_transformers import SentenceTransformer
from rank_bm25 import BM25Okapi

client = anthropic.Anthropic()
embedder = SentenceTransformer('all-MiniLM-L6-v2')

def add_context_to_chunk(document: str, chunk: str) -> str:
    """Use Claude to prepend situational context to a chunk."""
    response = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=200,
        messages=[{
            "role": "user",
            "content": (
                f"<document>\n{document}\n</document>\n\n"
                f"<chunk>\n{chunk}\n</chunk>\n\n"
                "Give a short context to situate this chunk "
                 "within the overall document. Be succinct."
            )
        }]
    )
    context = response.content[0].text
    return f"{context}\n\n{chunk}"

def build_index(documents: list[str], chunks_per_doc: list[list[str]]):
    contextualized_chunks = []
    for doc, chunks in zip(documents, chunks_per_doc):
        for chunk in chunks:
            ctx_chunk = add_context_to_chunk(doc, chunk)
            contextualized_chunks.append(ctx_chunk)

    # Semantic index
    embeddings = embedder.encode(contextualized_chunks)

    # BM25 lexical index
    tokenized = [c.split() for c in contextualized_chunks]
    bm25 = BM25Okapi(tokenized)

    return embeddings, bm25, contextualized_chunks

When RAG Isn't Enough: Agentic Search

The Anthropic course goes one step further, exploring agentic search — where Claude doesn't just passively receive retrieved chunks, but actively decides what to search for, evaluates the results, and iterates if needed.

In an agentic RAG setup, Claude is given the retrieval system as a tool. It can call that tool multiple times with different queries, reason about what it found, and decide when it has enough information to answer confidently. This transforms RAG from a one-shot retrieval system into a dynamic research loop.
